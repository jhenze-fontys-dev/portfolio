Generator note – PUT /symbols/{id} updategedrag
Wat we hebben ontdekt

Bij het updaten van een symbol via de gegenereerde endpoint
PUT /api/sql/sqlite.finance/symbols/{id}
krijgen we een 404 “symbols not found”, zelfs wanneer:

het symbool bestaat

GET /symbols/search?symbol=AAPL correct { id: 1, ... } teruggeeft

de URL {id} correct is (/symbols/1)

Oorzaak

De gegenereerde symbols.update controller gebruikt req.body.id om de row te vinden,
en negeert of overschrijft req.params.id.

Als id niet expliciet in de request body zit, zoekt de controller naar:

id = undefined of id = 0
→ resulteert in 404 not found

Swagger bevestigt dit gedrag:

het request body schema bevat een id veld

de controller verwacht dit veld impliciet

Gevolg

Frontend calls zoals:

PUT /symbols/1
{
  "symbol": "AAPL",
  "prices_updated_through": "2026-01-10"
}


falen, ondanks dat het pad-id correct is.

Huidige workaround (frontend)

Altijd id meesturen in de body:

{
  "id": 1,
  "symbol": "AAPL",
  "prices_updated_through": "2026-01-10"
}

Aanbevolen fix voor de generator (toekomst)
Optie A (beste, REST-correct)

In de controller:

const id = Number(req.params.id);


en negeer req.body.id volledig bij updates.

Optie B (acceptabel)

Als req.body.id ontbreekt, fallback naar req.params.id:

const id = req.body.id ?? Number(req.params.id);

Optie C (Swagger-consistent)

Verplicht id in request body en

documenteer duidelijk dat {id} in de URL cosmetisch is

(maar dit is minder REST-achtig en verwarrend voor frontend-implementaties)

Conclusie

De generator werkt functioneel, maar wijkt af van standaard REST-expectaties.
Voor betrouwbaarheid en DX is het sterk aan te raden om:

👉 req.params.id leidend te maken bij PUT /resource/{id}

Dit voorkomt stille 404’s en maakt frontend-helpers (zoals DailyPriceUpdater) eenvoudiger en robuuster.

er is een aanpassing gedaan in de sql driver

Generator notitie – SQL driver: Primary Key ID normalisatie
Context

In de generator-architectuur komt id vanuit de HTTP-laag (req.params.id) altijd binnen als string.
Dit is correct en gewenst voor generieke controllers (UUID’s, string PK’s, etc.).

Bij SQL-dialects (met name SQLite en Postgres) hebben tabellen echter vaak een
INTEGER PRIMARY KEY.

Probleem

De gegenereerde flow was:

controller (id: string)
→ service (id: string)
→ sqlDriver (raw query WHERE pk = :id)


Bij integer PK’s leidde dit tot:

SELECT/UPDATE/DELETE met "id" = "1" i.p.v. 1

geen match → affectedRows = 0

controller vertaalt dit naar 404 “not found”

endpoints als GET /symbols/1 en PUT /symbols/1 faalden, terwijl record bestond

Ontwerp-inzicht

Dit is geen fout in het accepteren van string-id’s in de controller.
Het probleem was dat niemand verantwoordelijk was voor type-normalisatie.

Beslissing (definitief ontwerp)

👉 De SQL driver is verantwoordelijk voor het normaliseren van primary key waarden.

Controllers en services blijven:

generiek

type-agnostic

dialect-onafhankelijk

De driver:

kent het model

kent de PK

kent het SQL-type

en kan dus veilig casten

Geïmplementeerde oplossing (driver)
Regel

Als de primary key kolom een integer-achtig type heeft
(INT, INTEGER, BIGINT, SERIAL, NUMBER)

Dan wordt de ontvangen id gecoerced naar Number

Anders blijft id onaangeroerd (UUID / TEXT PK’s)

Pseudocode
function coercePrimaryKeyId(model, id) {
  if (pkType is integer) {
    const n = Number(id);
    return Number.isFinite(n) ? n : id;
  }
  return id;
}


Toegepast in:

getById

update

remove

Voordelen

Fixt alle SQLite/Postgres integer-PK endpoints in één keer

Geen aanpassingen nodig in controllers of services

Veilig voor toekomstige UUID/string PK’s

Consistent met professionele ORM-architecturen

Generator-aanpassing (toekomst)

Deze logica moet worden:

verplaatst naar template.sqlDriver.hbs

toegepast op alle CRUD-methodes die een PK gebruiken

Principe:
Input-laag accepteert alles (strings/numbers), data-laag normaliseert op basis van schema.

Samenvattende one-liner

Controllers blijven generiek; SQL drivers normaliseren primary keys op basis van kolomtype.

📌 Notitie – SQL driver fix (UPDATE / DELETE affected rows)
Probleem

Bij PUT (update) en soms DELETE requests gaf de API 404 "not found", terwijl het record wél bestond in de database.

Dit kwam doordat:

De SQL driver controleerde of een update/delete “gelukt” was

Maar Sequelize geeft per dialect een andere return shape bij QueryTypes.UPDATE en QueryTypes.DELETE

Daardoor werd affectedRows soms verkeerd geïnterpreteerd als 0

De service gaf dan null terug → controller vertaalde dit naar 404

Oorzaak (technisch)

Sequelize returnt bij UPDATE/DELETE o.a.:

Dialect	Mogelijke return values
SQLite	[undefined, affectedCount]
SQLite	[affectedCount]
Postgres	affectedCount
Postgres	{ rowCount: affectedCount }

De generator ging impliciet uit van één vaste vorm, wat niet klopt.

Oplossing

In de SQL driver:

Een helper toegevoegd:

function extractAffectedCount(result) { ... }


Deze helper gebruikt in:

update()

remove()

Alleen null retourneren als affectedCount === 0

Hierdoor:

Correct gedrag bij alle SQL dialecten

Geen foutieve 404’s meer bij geldige updates/deletes

Controllers en services blijven ongewijzigd (goede scheiding van lagen)

Architectuurles (belangrijk)

✔️ Input-layer (HTTP) mag strings accepteren
✔️ Driver-layer moet dialectverschillen afhandelen
❌ Controllers/services mogen niet dialect-specifiek worden

👉 Deze fix hoort structureel in de generator template voor template.sqlDriver.hbs.

Actie voor volgende generator-update

Verplaats extractAffectedCount() + aangepaste checks naar de SQL driver template

Laat gegenereerde services/controllers ongewijzigd

Er zitten limmits in de driver... 


✅ In sql.driver.js (1 plek), maak de defaults instelbaar via .env en (optioneel) via query params in filters.

1) .env-instelbare defaults (super simpel)

Bovenin sql.driver.js voeg je toe:

const DEFAULT_LIMIT = Number(process.env.SQL_DEFAULT_LIMIT || 100);
const DEFAULT_SEARCH_LIMIT = Number(process.env.SQL_SEARCH_LIMIT || 50);
const MAX_LIMIT = Number(process.env.SQL_MAX_LIMIT || 1000);


En verander de function signatures naar:

async getAll({ model, limit = DEFAULT_LIMIT, offset = 0, orderBy, orderDir } = {}) {

async search({ model, filters, limit = DEFAULT_SEARCH_LIMIT, offset = 0 } = {}) {


En clamp daarna:

limit = Math.min(Number(limit) || DEFAULT_LIMIT, MAX_LIMIT);
offset = Number(offset) || 0;

2) Query override laten werken (zonder controller fix)

Omdat controllers req.query als filters doorgeven bij /search, kun je in search() limit/offset uit filters halen (zoals ik eerder stuurde). Dan werkt:

/daily_prices/search?symbol_id=1&limit=800&offset=0

Voor je “future plan” (generator)

Ja: ideaal is dat je generator in controllers:

getAll → req.query.limit/offset/orderBy/orderDir doorgeeft naar service

search → óók limit/offset (en evt orderBy/orderDir) doorgeeft als aparte options, niet als filters

Maar voor nu is de driver de meest pragmatische plek.

template controller aangepast:

Wat is er aangepast in de controller-template?

Query-parameters worden nu doorgegeven

limit, offset, orderBy, orderDir uit req.query worden expliciet gelezen en gevalideerd.

Deze opties worden doorgestuurd naar de service (getAll / search).

Geen hardcoded limiet meer in de controller

De controller bepaalt niet meer zelf “100” of “50”.

De uiteindelijke cap zit nu alleen in de driver (centrale plek).

search() ondersteunt pagination & sorting

/search gebruikt nu:

search(filters, options)


i.p.v. alleen search(filters)

Input-layer blijft generiek

IDs blijven strings (HTTP-laag correct).

Numerieke parsing gebeurt alleen voor pagination-velden.

Effect

Frontend kan nu grote datasets expliciet opvragen (limit=800)

Backend houdt centrale controle via de driver

Generator-architectuur blijft netjes gescheiden per laag

Dit is precies de juiste plek voor deze verantwoordelijkheid 👌
