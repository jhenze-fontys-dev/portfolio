# ADR-0003 — V2.1 Upgrade Scope and Technical Objectives

Status: Accepted
Version: 2.1 (Planned)
Owner: Fullstack Template Project
Date: 2025-12-27

## Context

V2.0 introduced a structured code-generation pipeline for backend artifacts
(models, services, controllers, routes, env, SQL adapters, docs, etc.).
During the development of V2.0, several tools also evolved organically
(reset tools, documentation tools, SQL testing tools, etc.).

As a result, a number of technical gaps and inconsistencies were identified
that should not be patched ad‑hoc, but instead addressed holistically in
a focused **V2.1 maintenance + architecture stabilization release**.

This ADR defines the scope and priorities for V2.1.

## Goals for V2.1

V2.1 is **not a feature release** — it is a **structural correctness & security hardening release**.

Primary goals:

- Strengthen consistency across generators and tools
- Improve security posture (especially SQL input handling)
- Cleanup naming, structure, and responsibilities
- Reduce hidden assumptions in tooling
- Make future extensions predictable and safe

## Out‑of‑Scope

V2.1 does **not** include:

- New domain features
- UI / frontend work
- Major refactors outside the generator + tooling layer

Anything outside structural correctness will be deferred to V2.2+.

## Decision / Work Areas

### 1) Reset Tooling Re‑Architecture

`reset:generated` must:

- ONLY delete generated artifacts
- NEVER implicitly trigger regeneration
- Leave the project in a clean “ready to generate” state

Additional expectations:

- Tree + Path docs **may** refresh after reset (configurable)
- Logging output stays clean, chronological, and readable
- `reset:db` becomes dialect‑aware and logged using the shared logger

---

### 2) Logger Normalization

A single shared logger is used across:

- generators
- tools
- testing utilities
- reset scripts

Standardized actions:

```
create | update | delete | rmdir | skip | run | report | info
```

No ambiguous `write` usage remains.

The logger is treated as **a shared tool, not a generator helper**.

---

### 3) Testing Tool Migration

The legacy `sqlScanner.cli.js` becomes:

```
dev/tools/testing/test.sqlInjection.js
```

Changes:

- Uses shared logger (header / footer / results)
- Produces reports in:

```
docs/backend/testing/sqlscan/
```

- Operates on the **OpenAPI manifest + per‑source spec**
- Supports future CI automation
- Output indexed in `index.json`

---

### 4) SQL‑Injection Findings — **Service‑Layer Remediation Policy** (NEW)

#### Problem

The SQL‑Injection test revealed that many generated service methods:

- accept raw parameters from controllers
- pass values directly into database query builders
- do **not validate type, range, format, or semantic rules**

Even when the driver prevents injected SQL execution,
this is still a **security and correctness failure**, because:

- invalid values reach the persistence layer
- domain rules are bypassed
- injection paths remain latent and repeatable

Example pattern identified in V2.0:

```js
async findById(id) {
  return db('Tasks').where({ id });
}
```

This accepts values such as:

- `"1 OR 1=1"`
- `"abc"`
- negative numbers
- non‑UUID values for UUID PKs

#### Root Cause

- Service layer was treated as a thin DB wrapper
- Validation was assumed to happen in controllers (often missing)
- Metadata did not define validation semantics
- The generator reproduced the weakness systematically

Therefore: **the issue is systemic, not local**.

#### Decision (V2.1)

All generated service methods must:

- validate inputs **before querying**
- reject invalid values using structured error helpers
- align validation rules with metadata definitions
- never rely on controller‑only validation

#### Enforcement Model

Validation will be:

- derived from metadata (types, constraints, PK rules)
- generated into reusable helpers
- called inside every service entry point

Target pattern:

```js
async findById(id) {
  const validId = validate.id('Tasks.id', id); // throws if invalid
  return db('Tasks').where({ id: validId });
}
```

#### Acceptance Criteria

- SQL‑Injection test reports show **zero high‑risk findings**
- Invalid values never reach persistence layer
- Metadata becomes the **single source of validation truth**
- New generators cannot emit unvalidated DB calls

Classification:

- Security hardening
- Architectural integrity improvement
- Backward‑compatible API behavior

---

### 5) DB Reset Improvements

- Dialects discovered from `.env`
- Future‑ready for **multiple DB instances per dialect**
- Move toward **metadata‑aware DB discovery** in V2.2
- Standardized `run` output formatting

---

### 6) Directory Structure Consistency

- Clear separation between:
  - `dev/generators`
  - `dev/tools`
- Testing utilities prefixed `test.*`
- No stray duplicate / legacy folder trees

---

## Risks

- Regeneration may modify many files at once
- Validation rules may expose hidden API misuse
- Tooling behavior changes require developer awareness

---

## Rollout Strategy

- Implement in a dedicated `v2.1` branch
- Regenerate backend artifacts
- Run:
  - functional tests
  - SQL‑Injection tests
- Freeze generator output when stable

---

## Status

Accepted for implementation as the first milestone after V2.0 completion.
