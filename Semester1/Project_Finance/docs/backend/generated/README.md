
# Generated Backend Documentation — Index

This document describes the structure and purpose of the **generated backend
documentation** under `docs/backend/generated/`.

It follows the shared documentation signature defined in `docs/style/docs-signature.md`.

---

## 1. Purpose

The `docs/backend/generated/` folder is the central location for all
**machine‑generated backend documentation artifacts**, including:

- OpenAPI specifications
- security test reports (SQL‑injection)
- (in future) human‑readable summaries of endpoints, models, routes, services

Everything under this folder is considered **safe to delete and regenerate**.

---

## 2. Structure

The target structure for generated backend docs is:

```text
docs/backend/generated/
  README.md          # this file (static index)
  openapi/           # machine-readable OpenAPI artifacts
    openapi.manifest.json
    sql/
      sqlite.<dbName>.openapi.json
      postgres.<dbName>.openapi.json
  sqlscan/           # JSON reports from SQL-injection tests
    sqlscan-*.json
  api.md             # (future) generated endpoint overview
  models.md          # (future) generated data model overview
  routes.md          # (future) generated routes overview
  services.md        # (future) generated services overview
```

Notes:

- `<dbName>` is the name of the database for a given source. In this template
  the example name is `fullstack_test`, but in your project it will match your
  own configuration.
- The `.md` files (`api.md`, `models.md`, etc.) will be generated by dedicated
  docs generators in a future version (V2.1+).

---

## 3. OpenAPI Artifacts

OpenAPI artifacts live under:

```text
docs/backend/generated/openapi/
```

and are managed by the `gen:openapi` step (or `gen:all`). They are used by:

- backend testing tools (e.g. SQL‑injection tester)
- future client generators
- any automation that needs a formal API contract

These files are **never edited by hand**.

---

## 4. SQL‑Injection Test Reports

The SQL‑injection test CLI writes reports into:

```text
docs/backend/generated/sqlscan/
  sqlscan-<timestamp>.json
  index.json
```

These JSON files capture:

- which endpoints were tested
- which payloads were sent
- how the API responded
- which potential findings were detected

They are designed to be both human‑inspectable and machine‑readable.

For more details, see `docs/backend/testing/README.md`.

---

## 5. Manual vs Generated Docs

Manual documentation should **not** be placed under `generated/`. Instead, use:

- `docs/backend/` for conceptual backend docs
- `docs/backend/manual/` for handwritten API notes

This separation ensures that:

- generators can safely reset `docs/backend/generated/`
- manual writing lives in stable locations

---

## 6. Future Work (V2.1+)

Planned future generators may include:

- `generate.apiDocs.js` → writes `api.md`
- `generate.modelDocs.js` → writes `models.md`
- `generate.routeDocs.js` → writes `routes.md`
- `generate.serviceDocs.js` → writes `services.md`

All of them will output into this folder, keeping generated documentation
centralized and consistent.
